# T1 — Critical Error Handling

Status: `done`

## Goal

After this tranche, the parent window receives an explicit error message when PDF generation fails, and the `pdfPending` mutex can never deadlock — even if pdfmake's internal `.download()` callback never fires. These are production bugs that can silently break the app for all users.

## Steps

### 1. Add `EXPORT_PDF_ERROR` message type

The parent needs a way to distinguish "still processing" from "crashed." Add an error response alongside the existing `EXPORT_PDF_DONE`.

### 2. Wrap `pdfMake.createPdf().download()` in a Promise with timeout

The current callback-based `.download()` has no fallback if the callback never fires. Wrap it in a Promise with a 30-second timeout so `pdfPending` always resets.

```ts
function downloadPdf(
  docDef: TDocumentDefinitions,
  filename: string,
): Promise<void> {
  return new Promise<void>((resolve, reject) => {
    const timeout = setTimeout(() => reject(new Error('PDF download timed out')), 30_000);
    try {
      pdfMake.createPdf(docDef, undefined, FONT_DICT, vfs)
        .download(filename.replace(/\.md$/, '.pdf'), () => {
          clearTimeout(timeout);
          resolve();
        });
    } catch (err) {
      clearTimeout(timeout);
      reject(err);
    }
  });
}
```

### 3. Rewrite the message handler's try/catch

Move `pdfPending = false` into a `finally` block so it always resets. Send `EXPORT_PDF_ERROR` in the catch block with the error message.

```ts
try {
  const docDef = await markdownToDocDef(filename, markdown);
  await downloadPdf(docDef, filename);
  source?.postMessage({ type: 'EXPORT_PDF_DONE' }, { targetOrigin: origin });
} catch (err) {
  console.error('[pdf-export] generation failed:', err);
  source?.postMessage(
    { type: 'EXPORT_PDF_ERROR', error: String(err) },
    { targetOrigin: origin },
  );
} finally {
  pdfPending = false;
}
```

### 4. Verify

- Trigger a PDF export with valid markdown — confirm `EXPORT_PDF_DONE` is received by the parent.
- Trigger a PDF export with malformed content that causes pdfmake to throw — confirm `EXPORT_PDF_ERROR` is received.
- After a failed export, trigger another export — confirm `pdfPending` was reset and the second export works.
- Verify the 30-second timeout by mocking a `.download()` that never calls back — confirm the handler rejects and `pdfPending` resets.

## Acceptance Criteria

- [ ] Parent window receives `{ type: 'EXPORT_PDF_ERROR', error: string }` when generation fails
- [ ] `pdfPending` resets to `false` in all code paths (success, error, timeout)
- [ ] A 30-second timeout rejects the download Promise if the callback never fires
- [ ] Console logs the error with `[pdf-export]` prefix for debugging
- [ ] No regression: successful PDF exports still trigger `EXPORT_PDF_DONE` and download the file
